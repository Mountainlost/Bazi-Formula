# 项目说明

## 项目定位
这是一个面向 `Windows + Python 3.10` 的八字工程化仓库，目标是提供可安装、可测试、可审计、可持续迭代的子平法单流派原型，而不是聊天式算命项目。
当前仓库定位是规则驱动原型，不是完整商业产品。

## 当前已实现层级
1. 真实四柱排盘
2. 十神基础映射
3. 旺衰评分 v0
4. 候选用神 v0
5. 最终用神受控版 v0（`final_useful_god_v0`）
6. 大运起运骨架
7. evidence 审计链

## 当前支持范围
1. `calendar_type=solar`
2. `calendar_type=lunar`，但只做进入既有排盘链路前的受控农历转公历
3. 时区白名单：`Asia/Shanghai`、`Asia/Singapore`、`Asia/Tokyo`
4. `true_solar_time=false`
5. `true_solar_time=true`，但仅支持 `configs/input_normalization.yaml` 中登记的出生地经度映射
6. 支持年份范围 `1900-2099`
7. 固定执行顺序：`calc -> judge -> report -> audit`

## 当前未实现
1. 完整格局体系
2. 调候体系
3. 神煞
4. 流年流月
5. 任意全球时区输入
6. 缺少闰月标识的闰月农历输入
7. 多流派混用

## 当前输出定位
- `chart.json`：真实四柱、大运起运骨架与排盘依据
- `rules.json`：十神、旺衰、候选用神、最终用神受控版等规则层结果
- `report.json`：只对 `chart.json` 与 `rules.json` 做结构化转述
- `audit.json`：对 evidence、字段边界、规则一致性和越界表达做审计

## 最终用神受控版 v0 的定位
- 它建立在候选层、旺衰层、月令偏置和五行分布之上
- 只有当前证据足够一致时才允许输出 `determined`
- 如果处于中和态、证据不足或存在冲突，必须返回 `insufficient_for_final_determination` 或 `blocked_by_conflict`
- 该层仍不是完整传统命理中的最终判定系统
- 当前仓库也不是完整商业产品

## 维护原则
- 不混入紫微、六爻、西占、塔罗等其他流派
- 不让解释层反向覆盖规则层
- 规则结论必须可回溯到本地映射表与 `evidence_refs`
- 在不破坏兼容性的前提下，优先保持中文业务目录与中文业务文档
